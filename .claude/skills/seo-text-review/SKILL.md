# SEO Text Review (конкурентное ревью текста под Яндекс)

Ревью текста страницы относительно топ-5 конкурентов в выдаче Яндекса по целевому
запросу. Цель — измеримо ответить на вопрос «дотягивает ли текст до уровня топа?»
и, если нет, дать предметные правки.

## Что подаётся на вход

- **URL страницы** пользователя (текст извлекается автоматически).
- **Целевой запрос Claude определяет сам из H1 страницы** — не спрашивает пользователя
  (см. Шаг 1). Пользователь может явно указать запрос рядом со ссылкой — тогда
  используй его, а H1 только для сверки.
- *(необязательно)* регион Яндекса (по умолчанию 213 — Москва, т.к. основной
  кейс — аптечная выдача по Москве и области), глубина топа (по умолчанию 5),
  порог прохождения (по умолчанию 85%).

Если пользователь даёт сразу текст вместо URL — работай с текстом, шаг извлечения
страницы пропусти (тогда запрос придётся уточнить, H1 неоткуда взять).

## Рабочий процесс

Выполняй строго по шагам. Не пропускай конкурентов: вся оценка строится на
сравнении с топ-5, без них цифры бессмысленны.

### Шаг 1. Извлечь и очистить текст целевой страницы
Используй `web_fetch` по URL пользователя с **большим лимитом токенов** (текст
инструкции у аптек часто идёт ниже навигации — при малом лимите он не попадёт в
выдачу). Сохрани сырой результат в `target_raw.txt`. Если `web_fetch` вернул
403/503/401 или страница за антиботом — возьми её через Claude в Chrome (реальный
браузер пользователя): `navigate` → `get_page_text` (см. Шаг 3, тот же приоритет
источников).

Затем очисти его от навигации/мусора:
```bash
python scripts/clean_text.py --in target_raw.txt --out target.txt
```
**Обязательно просмотри `target.txt` глазами** и вручную убери остаточный мусор
(мегаменю без переносов строк скрипт не разобьёт). В `target.txt` оставляй только
описательную прозу: описание, состав, показания, дозировку, противопоказания,
побочные и т.п. Оба файла понадобятся дальше: `target.txt` — для анализа прозы,
`target_raw.txt` — чтобы отличать «нет на странице вообще» от «есть в UI, но не в
тексте».

Если пользователь дал сразу текст вместо URL — это и есть `target.txt`,
`target_raw.txt` не будет (различение UI/проза тогда недоступно).

**Определи целевой запрос из H1.** Вытащи заголовок страницы:
```bash
python scripts/clean_text.py --in target_raw.txt --h1
```
Нормализуй H1 в поисковый запрос: убери гео- и брендовые «хвосты» и ценовой шум
(«в аптеках Москвы», «в Москве», «цена от 1075 руб», «интернет-аптека», «Таблетка.ру»,
«Ваша №1»), оставь предметное ядро (название препарата/категории). Для коммерческих
страниц добавь интент-модификатор «купить», если H1 явно про покупку, иначе оставь
ядро как есть. Примеры:
- H1 «Купить Детралекс в аптеках Москвы» → запрос «детралекс купить»
- H1 «Детралекс цена от 1075 руб» → запрос «детралекс купить»
- H1 «Витамин D3: для чего нужен» → запрос «витамин d3»

Этот запрос используй на всех следующих шагах. Если H1 пустой/нерелевантный —
выведи запрос из `<title>` и заметных ключей текста, и укажи в отчёте, как он получен.

### Шаг 2. Получить топ-5 конкурентов из Яндекса
Запусти `scripts/fetch_serp.py` (см. его docstring). Он читает ключ SERP API из
переменных окружения и возвращает JSON со списком URL топа.

```bash
python scripts/fetch_serp.py --query "<запрос>" --region 213 --top 5 --out serp.json
```

Провайдеры: `xmlriver` (по умолчанию) или `yandex_xml`. Креды — в env (скрипт
подскажет, какие именно, если их нет).

**Фолбэк при закрытой сети.** Песочница может блокировать домен SERP API
(в ответе egress-прокси будет `x-deny-reason`). Тогда:
1. сообщи пользователю, что нужно разрешить домен SERP API в настройках сети, **либо**
2. попроси вставить топ-5 URL выдачи вручную и продолжай с ними.

Никогда не выдумывай URL конкурентов из головы — это обесценивает анализ.

### Шаг 3. Извлечь и очистить тексты конкурентов
Для каждого из топ-5 URL извлеки основной текст. Источники по приоритету —
переходи к следующему только если предыдущий не дал чистый текст:

1. **`web_fetch`** (с большим лимитом токенов) — быстрее всего, пробуй первым.
2. **Сниппеты/пассажи из SERP API** — если в ответе `fetch_serp` есть `passages`/
   `title`, для gap-анализа ключей их часто достаточно.
3. **Claude в Chrome (реальный браузер пользователя)** — для площадок, которые
   отдают `web_fetch` 403/503/401 (eapteka, asna, gorzdrav и т.п.). Это честный
   доступ через живую сессию пользователя, не обход защиты. Требует подключённого
   расширения Chrome. Капчу не решай — если появилась, попроси пройти её вручную
   и продолжай. Темп умеренный, с паузами. Рабочий рецепт (проверен на аптеках):
   - `tabs_context_mcp(createIfEmpty=true)` — один раз, получить `tabId`;
   - `navigate` на URL, затем `wait` ~4 сек (антибот и рендер);
   - текст инструкции у аптек грузится **лениво и ниже карточки** — `get_page_text`
     обычно отдаёт только верхнюю `<article>`-карточку. Поэтому проскролль вниз
     2–3 раза (`computer` scroll, по ~10–12 тиков, с `wait` между ними), чтобы
     подгрузить блок инструкции;
   - вытащи прозу через `javascript_tool`: возьми `innerText` основного контейнера
     и **отрежь по смысловому якорю**, а не с начала страницы (там навигация).
     Якоря, которые надёжно попадают в описательную часть: `Фармакодинамика`,
     `Фармакокинетика`, `Показания`, `обладает венотониз`. Пример:
     `(()=>{const t=(document.querySelector('main')||document.body).innerText.replace(/\n{3,}/g,'\n\n');const i=t.search(/Фармакокинетика|Фармакодинамика|Показания/);return i>-1?t.slice(i,i+3500):t.slice(0,3500)})()`
   - вывод инструмента может обрезаться на ~1500 символов — при необходимости сделай
     ещё один срез по следующему якорю (напр. `Побочные`, `Противопоказания`), чтобы
     добрать остаток.
4. **Ручная вставка** — если всё выше недоступно, попроси пользователя вставить текст.

НИКОГДА не подменяй User-Agent под YandexBot и не маскируйся под поискового робота —
это обман сервера и обход бот-детекта (площадки и так верят rDNS-проверке). Реальный
браузер пользователя — да; поддельный краулер — нет.

Каждый полученный текст прогоняй через `scripts/clean_text.py` и просматривай.
Складывай очищенную прозу в `competitors/1.txt … 5.txt`. Если часть площадок так и
не открылась — работай с тем, что есть (минимум 3), и явно отметь в отчёте, сколько
конкурентов и каким источником получено.

### Шаг 4. Посчитать ключевые метрики
Запусти `scripts/keyword_stats.py` — он лемматизирует тексты `pymorphy3` и считает:
покрытие запроса, частотные таблицы, объёмы, и недобранные ключи (частые у топа,
отсутствующие у пользователя).

```bash
python scripts/keyword_stats.py --target target.txt --target-raw target_raw.txt \
    --competitors competitors/ --query "<запрос>" --out stats.json
```

Передавай `--target-raw` всегда, когда он есть: тогда каждый недобранный ключ
получает метку `where`:
- `absent` — термина нет на странице вообще → правка «добавить»;
- `ui_only` — термин есть в интерфейсе (цена, кнопка, фильтр), но не в прозе →
  правка «закрепить в тексте», а не «добавить»;
- `unknown` — `--target-raw` не передан, отличить нельзя.

Это нужно, чтобы не давать ложную правку «у вас нет цен», когда цены есть в
карточке. Семантику и интент оценивай сам, опираясь на тексты и `stats.json`.

### Шаг 5. Выставить оценку
Считай оценку строго по рубрике из `references/scoring.md` (4 оси, веса, формула).
Каждую ось оцени 0–100, выведи взвешенную итоговую оценку. **Покажи расчёт** —
пользователь должен видеть, из чего сложился балл, а не голую цифру.

### Шаг 6. Сформировать правки (если итог < порога)
Если итоговая оценка ниже порога (по умолчанию 85%) — дай конкретные,
исполнимые правки, привязанные к просевшим осям. Различай типы недобранных ключей:
для `absent` — «добавить блок/упоминание», для `ui_only` — «закрепить в
описательном тексте то, что сейчас только в карточке/кнопке». Не «улучшите текст»,
а «добавьте блок про X, он есть у 4 из 5 конкурентов и закрывает интент Y».

Если оценка ≥ порога — так и скажи, и дай 2–3 необязательные точки роста.

### Шаг 7. Подготовить переработанный текст с подсветкой
Если есть правки на добавление блоков — собери переработанный текст: исходный текст
с вставленными новыми блоками.

Правила:
- **Новые блоки пиши оригинальной прозой, своими словами.** НЕ копируй формулировки
  конкурентов — это и SEO-уникальность, и защита от плагиата. Передавай факты, а не
  чужие фразы. При этом **вплетай в блоки целевые ключи из списка недобранных** —
  чтобы заодно закрыть ось «ключи».
- Проверяй уникальность каждого блока против корпуса конкурентов:
  ```bash
  python scripts/uniqueness_check.py --new new_blocks.txt --competitors competitors/ --threshold 0.90
  ```
  Блоки со статусом FAIL переписывай оригинальнее, пока все не дадут OK.
- Вставляй блоки в логичные места по смыслу (ФК — рядом с механизмом, побочные — после
  противопоказаний).
- **Две подсветки в .docx:**
  - новый блок целиком — абзац с фоновой заливкой (`shading fill "FFF2CC"`, янтарь),
    можно с боковой полосой;
  - внутри блоков **отдельные внедрённые ключи** из рекомендованного списка — через
    **Word-маркер** (`TextRun highlight: "green"`), а не заливку: маркер виден во всех
    просмотрщиках, включая мобильные и предпросмотр, и это явно другой цвет;
  - исходный текст не трогай и не подсвечивай.
  Добавь легенду с расшифровкой обоих цветов.
- **YMYL:** медблоки (побочные, фармакокинетика, дозировки) — строго по официальной
  инструкции; отзывы не фабрикуй (структурный блок-приглашение, не фейковые тексты).

### Шаг 8. Собрать единый итоговый документ .docx
Финальный файл — **один документ из двух частей**, используя навык работы с .docx
(`/mnt/skills/public/docx/SKILL.md`):
- **Часть 1. Анализ** — отчёт по шаблону ниже (оценка, оси, пробелы, рекомендации).
- (разрыв страницы)
- **Часть 2. Переработанный текст** — результат из Шага 7 с двумя подсветками и легендой.

Сохрани в `/mnt/user-data/outputs/` и покажи через `present_files`.

## Структура итогового документа (.docx)

Соблюдай порядок разделов.

**Часть 1. Анализ:**
```
# SEO-ревью текста: <запрос>
Дата, URL, регион, какие конкуренты и каким источником получены.

## Итоговая оценка: NN% — [Проходит / Не проходит порог 85%]
Таблица по осям: Ось | Вес | Балл 0–100 | Вклад в итог.

## Соответствие запросу и интенту
## Полнота тематического покрытия — подтемы топа; чего не хватает.
## Ключевые слова — покрытие запроса, недобранные ключи (таблица: ключ | у конкурентов | у вас).
## Структура и объём — объём vs медиана топа.
## Рекомендованные правки   <- только если итог < порога
```

**Часть 2. Переработанный текст** (на новой странице):
```
# Часть 2. Переработанный текст
Легенда: [янтарь] новые блоки  ·  [зелёный] внедрённые ключи из рекомендаций.
Исходный текст без заливки + новые блоки с подсветкой (две цветовые схемы).
```

## YMYL / аптечная специфика

Это коммерческая SEO-задача для легитимной аптеки. При этом контент медицинский
(YMYL), поэтому в правках:
- не предлагай добавлять медицинские утверждения, обещания эффекта, дозировки или
  показания, которых нет в первоисточниках/инструкции;
- рекомендации по E-E-A-T формулируй нейтрально (указать источники, экспертизу,
  ссылки на инструкцию), без сочинённых медицинских фактов.

## Файлы скилла

- `scripts/fetch_serp.py` — топ выдачи Яндекса через SERP API (XMLRiver / Yandex XML).
- `scripts/clean_text.py` — очистка извлечённого текста от навигации/UI перед анализом.
- `scripts/keyword_stats.py` — лемматизация и метрики ключей/полноты (`pymorphy3`),
  с различением `ui_only` / `absent` при наличии `--target-raw`.
- `scripts/uniqueness_check.py` — шинговая проверка уникальности новых блоков против
  корпуса конкурентов (порог 90%).
- `references/scoring.md` — рубрика оценки: оси, веса, формула, пороги.

Перед расчётом оценки всегда читай `references/scoring.md`.
